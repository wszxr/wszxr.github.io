5，设置文件的权限

  • chmod u/g/o/a +/- rwx:
    • chmod代表changemode指令
    • u/g/o/a分别代表给拥有者，拥有组，其他用户，所有用户
    • +/-代表增加，减少权限
    • +r代表增加读权限，+w代表增加写权限，x代表执行权限，可以+wrx等等
  • 可以用数字表示
    • rwx= 4+2+1 = 7
    • rw-= 4+2 = 6
    • r-x= 4+1 = 5
    • r--= 4
    • ---= 0
  • chmod 765 表示u 为7(rwx),g为6(rw-),0为5(r-x)

软链接，硬连接

1，ln -s 源文件 目标文件，可以建立软链接，目标文件相当于一个源文件的快捷方式

  • 软链接的文件相当于一个路径，指向文件名

  • 源文件最好使用绝对路径，不然目标文件变动后，找不到源文件

  • 修改目标文件的内容会同步到源文件

  • 源文件移动后，软链接会断开

2，linux文件系统，文件名和数据完全分开

  • 路径层，通过路径解析可以找到目录层的指定目录
  • 目录层，目录层存放文件名和iNode的映射
  • inode层，通过iNode中的元数据找到真实数据存的地方
  • 软链接相当于一个路径，指向一个目录，当目录中文件名变了，软链接就断了
  • 删除原文件也只是删除了源文件的目录层，iNode和数据都还在，所以对硬链接没影响，由于没有目录，路径就解析不到目录，所以软链接就断开

2，ln 源文件 目标文件，建立硬连接

  • 源文件移动或者

  • 硬连接相当于在目录新增一个文件名，和原来的文件名映射同一个iNode，就算原来的文件名删除，文件数据也依旧可以通过这个文件名访问

  • 当没有文件名映射到这个iNode的时候，iNode元数据指向的真实数据也会被删除

1，@RequestMapping的Value代表路径，同一个应用的路径不可以一样，不同的类中控制器路径也不能一样，一般设置一个修饰类的@RequestMapping可以解决，类上的路径设置为和页面相关

2，mvc：

  • m代表Model，包括service层和dao层
  • v是view，视图，比如jsp，html；控制器的返回值为string，配置的视图解析器会通过配置好的前缀和后缀拼接返回值，结果是一个路径，一般是一个本地的jsp或者html文件；视图解析器会解析文件展示到浏览器成一个页面
  • c是控制器，springmvc中是每个controller；javaweb中是servlet

3，Restful风格代码

  • 路径占位符 http://localhost:8080/api/login/{username}/{password}；通过大括号占位接收参数
  • 使用@Pathvariable(“username”) String uname来给方法参数赋值；@Pathvariable有两个属性，name和value，是同样的效果，指定参数在大括号中的名字
  • 不同请求方式方法的使用场景
    • get：查
    • post：增；非常规参数，比如文件，音频等都是通过post传输
    • put：修改
    • delete：删除
    • headers：请求头

4，状态码

  • 400：前端参数和后端params指定的对象没完全对应上
  • 404：前端的请求头和后端的headers指定的不一样那个

8，对密码进行md5加密处理

  • md5加密会生成一个32位的字符串
  • md5加密是单向不可逆的，不能通过结果反推原本的密码，因此在比对的时候，先进行加密在与数据库中加密好的数据进行比对即可

  • digestutils是基于springframework的，用于加密的工具类

9，后端生成接口文档，接口测试，knife4j

1，knife4j是一个框架，集成了swagger，简化了swagger的操作

2，在springboot中使用knife4j

  • 导入依赖
  • 配置类中创建需要的bean
  • 设置静态资源映射，把访问地址映射到静态资源上；如果没有设置映射，springmvc会把这个路径当做一个controller的路径，就找不到会404
  • 通过服务的ip+端口+/doc.html可以直接访问主页

3，knife4j常用注解

  • @Api：用在类上，比如controller类，描述类的作用、
    • tags参数写描述信息
  • @ApiModel：用在实体类上，比如dto，vo，entity，描述作用
    • description参数写描述信息
  • @ApiModelProperties：用在实体类的属性上
    • value参数写描述信息
  • @ApiOperation：用在方法上，描述方法的信息，作用，参数，返回值等
    • value参数写描述信息
  • @ApiImplicitParams：用于描述参数信息，可以包含一个参数列表
  • @ApiImplicitParam：描述参数列表中的每一个参数
    • name：必填，和方法签名中的参数名必须一致
    • value：非必填，对参数的描述
    • required：非必填，描述参数是否可以不传
    • paramType：必填，指定参数所在的位置
      • query：代表在路径中通过？拼接的参数
      • path：代表路径中使用{}restfull风格的参数
      • header：请求头中的参数
      • cookie：cookie中的参数
      • body：请求体中的参数
      • cookie和header都可以通过httprequest对象获取

15，获取用户上下文信息

1，在进行增删改操作时，要获取用户信息并存入记录中，以记录是谁进行的修改

2，方案一：在拦截器中，通过token反推出存入token的信息，再去库里查询完整的信息，存入threadlocal

  • 创建threadLocal实例，一般都会对threadlocal进行包装，提供静态的存取删方法
  • 通过请求头中的token，获取信息，再去库查询完整信息，再存入ThreadLocal
  • 注意请求结束后清空内容
  • 由于一个请求在controller，service，mapper直至返回都是同一个线程，所以这种方式在3层都可以获取

3，方案二：自定义一个注解，例如@Login，注解类型为修饰参数，例如User user

  • 在参数解析器中，取到有@Login修饰且参数类型是User的参数
  • 通过token和库查到用户信息
  • 将用户信息通过反射插入该参数
  • 这时参数就有用户信息，在每一层方法都带上此参数，就能获取到用户信息
  • 参数解析器和拦截器都是在controller之前执行，但是分工不一样，应正确使用
  • 这种方法只能给controller的参数赋值，如果其他层想用，那么将此参数作为参数传递下去即可

16，mybatis中带条件的分页查询

1，传入的数据为，需要展示的页码，每一页的数量，查询条件；前端的数据应当专门建一个pageXXXDTO来接收

  • 注意前端的的参数是quey类型(路径中拼接)，后端使用对象可以直接接收；如果前端传入的是JSON，那么后端要加@RequestBody

2，分页的结果应当封装到一个pageResult类里，类的属性根据前端需要的属性来定，正常需要的是总条数，和当前展示页的数据列表

3，一般返回前端的都是Result统一结果，一般包含响应信息，响应码，data数据；这个data一般用T泛型表示或者是Object，所以返回的时候返回Result<pageResult> 就指明了data类型是pageResult，然后pageResult又包含总条数和数据列表

4，手动方案进行分页

  • 手动先根据条件筛选，然后计算需要页码的开始和结束条目，使用limit截取返回即可
  • start计算的时候，page一定要减一，比如需要第一页的，那就从0*pageSize=0开始；第二页就是从1*pageSize=pageSize开始

5,使用插件进行分页，原理就是，把pageNum，pageSize保存到threadLocal中，在sql执行之前，拦截执行sql的方法，判断threadLocal是否有分页信息，有的话就进行拼接limit然后执行

  • 在mapper接口中，如果安装了mybatisx插件，mapper接口没有对应sql会爆红，alt+enter可以自动创建sql，自动填写id和返回值
  • 必须导入pagehelper依赖才可以使用

30，Java时间日期

1，获取当前日期的指定格式

2，两种日期api的区别，新api的好处，什么情况用新的什么情况用旧的，除安全性外，效率是否有区别

  • 新api主要解决了simpledateformat的线程安全问题，simpledateformat无论是parse还是format方法，都会先使用calendar对象存中间状态，多个线程同时调用一个simpledateformat的方法时，会导致calendar对象出现线程安全问题，导致转换后的内容和原本预期的内容不一致；效率也是新api更好

3，前后端日期类型的参数传递

  • 前端传入指定的字符串格式 “2025:05:20”
  • 后端直接使用date接收会报错，格式转换错误
    • 可以在接收的属性上加spring的注解，@DateTimeFormat(pattern = “yyyy-MM-dd”) private Date createDate;
    • 可以对date类型做全局配置，spring.mvc.format.date=yyyy-MM-dd；spring.jackson.date-format=yyyy-MM-dd；spring.jackson.time-zone=GMT+8；参数传递需要指定springmvc的转换器，json传递需要指定Jackson的转换器
    • 也可以接收字符串，然后手动转成date类型
  • 服务器的时间保存到数据库
    • 数据库保存日期时间用datetime，保存时间用time；需要自动时区转换用timestamp
    • java的date和localdatetime可以同时对应datetime和time，instant和timestamp可以对应timestamp
    • 数据库datetime类型存什么就取什么，不管时区的事；timestamp会把时间转换成utc时区存入，所以存之前要指定时区，取的时候也要指定时区，否则可能导致时间错误
    • timestamp取的时候会自动以服务器时区为时区转换时间，因此在国际业务中非常好用，不用每次都手动根据当前时区转换，但是timestamp是从1970-2038，存历史事件等的日期存不进去
