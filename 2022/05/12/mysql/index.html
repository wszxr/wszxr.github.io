mysql

1,事务

事务操作

1，mysql事务默认自动提交，可以手动设置提交方式

查看事务提交方式，select@@autocommit;

@@autocommit=1时代表自动提交，

可以手动设置，set @@autocommit=0;改为手动控制事务

开启事务：start transaction/begin

提交：commit

回滚：rollback

事务特性

1，原子性(Atomicity)：事务是不可分割的最小单位，同时成功失败

2，一致性(Consistency)：事务完成时是所有的数据都保持一致状态

3，隔离性(Isolation)：隔离机制保证不受外在并发影响

4，持久性(Durability)：事务一旦提交或者是回滚，对数据的改变就是永久的,持久化到磁盘当中

并发事务

1，脏读：一个事务读取到另一个事务还未提交的数据

2，不可重复读：同一个事务查询同一行两次，两次的结果不一样

3，幻读：查询时没查到这条数据，插入时却已存在，好像是有个幻影

事务隔离级别

解决并发事务引发的问题。

1，读未提交（三种问题都可能出现)

2，读已提交(解决脏读问题)

3，可重复读(解决脏读不可重复读，mysql默认)

4，串行化，全部解决

2，存储引擎

mysql体系结构

  • 注意索引是在存储引擎层确定，也就是说不同的引擎的索引是不一样的。

存储引擎简介

1，是存储数据，建立索引，更新/查询数据等技术的实现

2，单位是表，每张表都可以单独指定存储引擎，所以也被叫做表类型

3，Mysql在V5.1之前默认存储引擎是MyISAM；在此之后默认存储引擎是InnoDB，不指定就是默认的

  • show create table 表名;可以查看建表语句
  • show engines;可以查看所有引擎。

4，创建表的时候直接指定就行

存储引擎特点

1,innodb:

特点：

  • 支持事务，DML（增删改)遵循ACID(事务四大特性)模型
  • 支持行级锁
  • 支持外键约束

总结：事务，外键，行级锁

文件：

  • innodb_file_per_table默认打开，表示每一张表都有自己的表空间文件
  • 每一张innodb表对应一个.ibd(表空间文件)，存储数据，表结构，索引。

2，myisam

1，是早期mysql默认引擎

2，特点：

  • 不支持事务和外键
  • 不支持行锁，支持表锁
  • 访问速度快

3，文件：

  • .sdi：存储表结构信息
  • .MYD：存储数据
  • .MYI：存储索引

3，Memory

1，存储在内存中，断电就会消失，只适合存储临时表或者是缓存

2，特点：

  • 内存中，速度快
  • 支持hash索引

文件：

.sdi

3，存储引擎选择

innodb(绝大部分场景)

对数据完整性要求高，有较多的并发增删改操作，只能用这个，因为只有他支持事务

myisam(被mongodb替代)

对数据完整性要求不高，增删改操作少，例如：日志，点赞，评论，这种少几条数据没关系的，用这个很合适，会比innodb快一些

memory(被redis替代)

常用于缓存，临时表

4，索引

概述

索引结构

  • 支持情况

问题：为什么采用b+tree？

1，相比于二叉树，层级更少查询效率高

2，相比于btree，btree叶子节点和非叶子节点都会保存数据，这样就导致每一页存的节点的量比b+tree少，那么在相同数据量下，btree的高度会高于b+tree，查询效率较慢。

3，相比于hash索引，hash索引不支持范围匹配，也不支持排序

索引分类

1，主键索引，默认创建且只能有一个

2，唯一索引，字段是UNIQUE时会创建索引

3，常规索引，快速定位

4，全文索引，查找文本中的关键字

innodb

  • 聚集索引，只能有一个，b+tree的叶子节点保存了整行的数据
  • 二级索引(非聚集索引)：可以有多个，叶子节点存的是每一行的主键id
  • 如果有主键，那么主键索引就是聚集索引
  • 没有主键，把第一个UNIQUE索引作为聚集索引
  • 没有主键也没有UNIQUE索引，innodb会自动生成一个rowid作为隐藏聚集索引
回表查询

  • 非聚集索引查到行id时，会根据这个id去查聚集索引以获得行数据
问题

答：第一个

  • 第一个需要行数据，直接走id也就是聚集索引就可以拿到
  • 第二个走二级索引，查到了id后还得回表查询id索引去获得行数据

索引使用

最左前缀法则
  • 联合索引查询必须从索引中最左边的列必须存在，否则索引全部失效
  • 中间的列不可跳过，跳过之后部分失效(此列右边的全部列的索引都将失效)
索引失效

1，不要在索引列上进行函数操作，不然索引会失效

2，字符串不加单引号会导致那一行索引失效

3，前面的模糊查询(%在最前面)会导致索引失效,包括前后都有%也会失效

4，or用于条件分割，只有当所有的条件列都有索引的情况下，索引才会生效，有一列没有索引，那么所有索引都会失效

5，数据分布影响，mysql会判断走索引和不走索引那个更快，例如is null,is not null

6，范围查询

范围查询右边的列索引失效，尽量使用>=,<=

sql提示

定义：在sql中加一些认为的提示，来达到优化的目的

  • use index,ignore index,force index
覆盖索引

虽然不是通过聚集索引查询，但是二级索引的结构中包含了要查询到的所有信息，就不需要回表查询了，就叫覆盖索引。

尽量别使用select*，极高可能会产生回表查询，

  • 答：username和password建立联合索引即可，不会产生回表查询
前缀索引

如果字符串很大的情况，把字符串存进b+tree中会耗费大量磁盘io，可以把字符串的一部分拿出来存进索引中，
