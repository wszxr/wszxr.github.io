JVM

1，内存结构

程序计数器（pc Register）

  1. java运行流程：
    • java源码–>jvm指令–>解释器–>机器码–>cpu执行
  2. 计数器作用：记录下一条指令的地址
  3. 特点：
    • java把寄存器当作程序计数器，是唯一不会出现内存溢出的问题的区域
    • 线程私有，每一个线程有自己独立的程序计数器

虚拟机栈（jvm Stacks）

  1. 定义：
    • 每个线程运行所需要的运行空间。
  2. 组成：
    • 由若干栈帧组成，每个栈帧对应一个方法的内存空间（局部变量，参数，返回地址···）。
    • 每个栈只有一个活动栈帧(顶部)，对应正在运行的那个方法。
  3. 栈内存溢出：
    • 单个栈帧内存过大(一般不会出现)。
    • 栈帧过多(递归时没有定义结束条件时)。
  4. 注意：
    • 垃圾回收不涉及栈内存，方法运行完自动释放。
    • 单个栈内存并不是越大越好，越大，能创建的线程数就越小。
  5. 方法中的局部变量是否有线程安全问题：
    • 静态的变量是共享的，线程不安全。
    • 通过传递参数进来的是不安全的。
    • 逃离出方法作用范围的变量不安全，例如return 出去的变量。
    • 方法内非静态基本类型变量是线程安全的，每个线程都有独立的一份。

本地方法栈（Native Method Stacks）

  1. Java中本地方法调用时使用的内存。

堆（Heap）

  • 通过new关键字创建的对象会使用堆内存。
  • 堆是线程共享的。
  • 堆中有垃圾回收机制.
  1. 堆内存溢出：
    • 当对象没有被任何一个引用所引用，jvm才会把它当作垃圾。
    • OutOfMemoryError。

方法区（Method Area)

  1. 定义（jvm1.8）：

    • 是一种规范，存储和类的结构相关的信息，运行时常量池。
    • 在虚拟机启动时被创建.
    • 在逻辑上其实是堆的一部分,可以不遵守此规范（由具体jvm实现决定）。
      • 例如：jvm1.8以下是在永久代里边
      • jvm1.8以上是在本地内存中的元空间里

  2. 溢出：

    • jvm1.8以前是永久代溢出。
    • 1.8以后是元空间溢出，元空间属于操作系统的空间。
    • 溢出场景：
      • spring和mybaties用cglib创建动态代理类。

  3. 常量池：

    • 就是在字节码文件中的一张表。
    • 作用，给指令一个符号，用于虚拟机寻找要执行的东西。

  4. 运行时常量池：

    • 常量池在字节码文件运行时加载到内存中的运行时常量池中，这时候符号变成了真实的内存地址。

    • StringTable（字符串常量池）:

      • 是一个hashtable结构而且不可以扩容。

      • 当第一次定义字符串时，如果串池中没有则会把对象放入串池中，如果有则直接使用串池中的对象。

      • 示例

      • /**
 * 关于串池的题
 */
public static void main(String[] args) {
    String s1 = "a";
    String s2 = "b";
    //常量拼接
    String s3 = "a"+"b";

    //s4,变量拼接实际上是新建一个空的StringBuild(),然后.append("a").append("b").toString
    //toString会把拼接好的值放到一个新的对象中，然后复制给s4
    String s4 = s1+s2;


    String s5 = "ab";

    //尝试将字符串对象放入串池，如果串池里面有相同的常量，就不加入，
    // 如果没有，就加入（jdk1.8以下会拷贝一份再加入拷贝的对象，拷贝出来的对象和原对象不一样），
    //无论加入或者是没加入都会返回串池中字符串常量对应的对象
    String s6 = s4.intern();

    System.out.println(s3==s4);//false =>             false=>//s3的对象在串池中，s4的在堆里面
    System.out.println(s3==s5);//true  =>正确答案      true=>因为String是不可变的，而且都是常量，因此结果是确定的，在编译期间就优化成"ab"
    System.out.println(s3==s6);//true  =>             true
}

      • 题目总结：

        • 当串池中没有时，新建一个String对象会加入串池，之后创建的字符串常量如果是一样的，那么就统一用串池中的那个对象。
        • 字符串常量对象拼接会在编译期间优化成拼接后的常量；而字符串变量拼接使用的是StringBuild，产生的对象存在堆中。
        • intern()方法会尝试将堆中的对象放入常量池，串池中有此常量，那就不加入，没有则加入(java1.8)；无论加入成功或者是失败方法返回值都是串池中对应常量的对象。

      • StringTable的位置：

        • java1.7的StringTable从永久代转到了堆空间中，因为永久代只有fullGC时才会执行回收。

      • StringTable也会被垃圾回收。

      • StringTable调优：

        • StringTable实际上是一个hashTable，根据实际情况，调整桶的个数就是调优了，桶的个数的最小值是1009。

直接内存(DirectMemory)

  1. 定义：

    • 常用于NIO操作，作为数据缓冲区。
    • 分配和回收此内存空间成本较高，但是读写性能高。
    • 不受jvm回收管理，因为是操作系统的内存。

    • 直接内存，java代码可以直接访问，比原始io少了一次缓存复制的过程。

  4. 直接内存的分配和回收都由unsafe对象完成：

    • 以ByteBuffer为例：它的构造器中获取了unsafe实例，并且调用unsafe实例的allocatememory（）和setmemory（）两个方法分配了直接内存。然后还使用Cleaner（虚引用对象）监测ByteBuffer实例，当ByteBuffer被GC回收时，会在Reference Handler线程中调用Cleaner的clean(),进而调用clean()中的unsafe.freeMemory()方法对直接内存进行释放。
    • java中默认禁止了显式GC，因为System.gc()是fullGC，会使得程序停留时间过长。
    • 这时候只能等待jvm进行GC，或者是直接使用unsafe对象的.freeMemory()方法。
    • Unsafe只能通过反射强制获取实例。

2，垃圾回收

如何判断垃圾可以被回收？

引用计数法

  • 当被一个对象引用，计数加一，取消引用则减一

  • 可能会出现循环引用，即两个对象除了互相引用外没有任何其它变量引用他们，无法使计数器归零，造成内存泄漏。

可达性分析算法（java中使用）

  • 判断是否直接或者间接被根对象引用，不是的话就标记

  • 根对象就是无论如何都不会被回收的一系列对象，例如：

    • System Class（java启动时的系统类）

  • 操作系统运行时引用的java对象

    • 被线程上锁的对象

  • 栈帧内引用的对象

并发的可达性算法

我们知道执行可达性分析的时候是要阻塞用户线程的，当堆空间的对象过多时，可达性算法的扫描将会使得程序停顿很久，我们可以通过并发执行来提高效率。

然而，在并发分析的时候，用户线程可能更改了某些对象的状态，所以并发执行可能带来两个问题。

  • 一是可能把应该标记为垃圾的对象错误标记为存活，这种情况并无大碍，下次再回收掉即可。
  • 二是把该存活的对象给标记为垃圾回收了，这样的结果是灾难的。

解决方法如下：

  • 增量更新法：

  • 如果经扫描的确认为安全（黑色）的对象在可达性分析期间被用户线程引用了一个未经扫描的对象（白色），那么会把此黑色对象置为灰色（已经被扫描过但是此对象上还存在没有被扫描过的引用），把这个引用记录下来，并发扫描结束后再把此灰色对象当作一个根对象扫描。

  • 简单说就是黑色新引用了白色，黑色就会变成灰色。

  • 原始快照法：

  • 如果并发中灰色对象切断了白色对象的引用，会把此引用记录下来，并在结束后把该灰色对象当作root对象重新扫描一遍。

  • 简单来说就是在开始分析的那一刻的快照被记录下来，先扫描那个快照，后边扫描并发中变化了的就可以了。

生存还是死亡（有复活机会，但只有一次）？

这个话题分两种情况讨论，

  • 首先，需要两次标记才会确认对象要被回收，第一次标记是在检查到没有和GCroot相连接的引用链时。

  • 第一种情况：如果对象被判断没有必要执行finalize（）方法，比如没有覆盖过finalize（）方法或者说finalize方法已经被调用过了（此方法只能被调用一次），那么就直接再次标记为可回收状态。

  • 第二种情况：对象被判断要执行finalize（）方法，比如对象覆盖过finalize（）方法，那么第一次标记之后，会进入F-Queue队列；然后由一个优先级很低的Finalizer线程来调用（只保证开始执行，不保证执行结束）队列中对象的finalize（）方法，在此期间，如果对象自救(再次被变量引用，例如在finalize()中把this赋值给某变量)，那么被标记的对象就会”复活“,复活后的对象的finalize（）方法将不能再被调用；如果没复活，那应该就真的该“死”（被回收）了。

  • 事实上，不要使用finalize()；深入理解jvm这本书上甚至建议我们忘了finalize()这个方法······哈哈哈哈哈哈！！！

四种引用

  • 强引用 ：只有当root对象都没有通过强引用引用该对象时，该对象才能被垃圾回收。

  • 软引用：只被软引用引用的对象，在一次垃圾回收之后，内存还是不够的情况下，就再次回收把被软引用的对象也回收掉。可以配合引用队列释放引用自身的内存。

  • 弱引用：只被弱引用引用的对象，只要发生垃圾回收，就会被当作垃圾回收掉。可以配合引用队列释放引用自身的内存。

  • 虚引用：为一个对象设置虚引用的唯一目的就是当这个对象被回收时收到一个系统通知，当虚引用引用的对象被回收时，虚引用会进入引用队列，由References Handler线程调用相关方法释放内存，例如Cleaner和ByteBuffer。必须配合引用队列使用。

  • 应用：软引用

    • 可以使用软引用和弱引用引用不太重要的资源（读取图片等），当内存不够时，会自动回收。
    • SoftReference a = new SoftReference<>(t1);这样就可以使得a软引用t1线程对象。a.get()可以获得被引用的对象。
    • 我们可以定义一个软引用队列作为SoftReference的参数，即可关联。也可以手动移除队列中的无用软引用对象。
      • //引用队列
        ReferenceQueue queue = new ReferenceQueue<>();
        //关联
        SoftReference a = new SoftReference<>(t1,queue);
      • t1被回收时，SoftReference入队列，t1回收完，SoftReference为空，我们人为的从队列取出，由gc回收。

  • 应用：弱引用：

    • 和软引用差不多，SoftReference换成WeakReference就可以了。自身的释放也需要配合引用队列。

垃圾回收算法

  • 都起源于分代收集理论
  • 分代导致了多种gc方式，
  • minorgc只回收新生代，majorgc只回收老年代（仅cms收集器实现）mixedgc收集全部新生代和部分老年代（仅G1收集器有）

1，标记清除算法

先标记可以释放的对象空间，后清除，这里的清除指的是记录下来被释放的内存的开始地址和结束地址，使他们可以再次被分配。

  • 优点，只需要记录开始和结束地址即可，速度很快。
  • 缺点
    • 执行效率不稳定，随着标记的增加或者是减少，效率也对应减少或者是增加。
    • 会产生较多的内存碎片，这也内存在被分配的时候不好处理。

2，标记整理

  • 先标记，后整理。整理的过程是把正常对象向内存的一端移动，占满内存碎片。
    • 优点，不会产生内存碎片
    • 缺点，对象移动，对应的内存地址也变了，效率低，而且移动的时候需要阻塞所有的用户线程。

3，标记，复制算法

  • 把内存均分为from区和to区两个区域，to区保持为空；步骤：
    • 把from区的正常对象复制到to区
    • 清空from区域
    • 调换from区域和to区域
    • 优点，解决了标记清除算法标记过多时效率低下。
    • 缺点，需要使用两倍的内存空间

Apple式回收

步骤：

  1. 新建对象会放入伊甸园中，当伊甸园不足以放下此对象时，触发一次minorGc，
  2. minorGc会采用标记复制算法，把伊甸园区域和from区域的正常对象复制到to中，然后清除伊甸园的数据，交换from和to，并且复制过来的对象年龄加一，
  3. 对象的年龄存储在对象头中，由四位存储，因此最大是15岁，超过15岁将会晋升至老年代区域，
  4. 老年代不足时会先触发minorGC,后触发fullGC。
  5. fullgc也清不下空间，就outofmemory错误。
    • 注意，一个线程的outofmemory错误不会导致整个java程序结束。
  • 注意，minorGC和fullGC都会stop the world（停止其他所有用户线程），因为垃圾回收会牵扯到对象内存地址的变化，这时候其他线程在使用的话会出错。
  • 伊甸园：form：to一般是8：1：1
  • fullGc比minorGC时间要长。
  • 当大对象（就算是清空新生代的伊甸园也放不下）存入时，不会触发垃圾回收，会直接进入老年代。

跨代引用假说

  • 当老年代的对象引用了新生代的对象时，minorGc就无法清除新生代的此对象，这时我们必须扫描新生代和老年代所有的对象用以判断该对象是不是可以被回收，显然这很不科学。

  • 跨代引用假说，等到新生代的此对象随着年龄的增加晋升到老年代中，就不存在跨代引用了。为了实现：

    • 我们在新生代创建一个叫做”记忆集“的全局数据结构，这个结构把老年代划分成了很多小块，只有包含了跨代引用的内存块才会和新生代对象一起被加入到GCRoots进行扫描。

  • HotSpot的实现

    • 使用卡表实现数据集(相当于HashMap和Map的关系)，在HotSpot中，卡表是一个字节数组。卡表中的每个元素对应其标识内存区域的一块特定大小的内存。这一块内存叫卡页。HotSpot的卡页是512字节。只要一个卡页中有一个对象存在跨代引用，就将对应卡表的数组元素的值标识为1，称为元素变脏。没有就为0。

    • CARD_TABLE [this address >> 9] = DIRTY;

    • 在垃圾收集的时候，判断卡页是否变脏，脏了就加入gcroots一并扫描。

  • 关于如何使得对象被引用时同时使得对应的卡页变脏？

    • 使用了写屏障，具体就是aop环绕编程，在引用变量赋值操作的时候就进行卡表的更新。

  • 虚共享问题，java操作单位是缓存行，当两个对象在同一个缓存行中并发操作，就会导致虚共享

    • 解决if (CARD_TABLE [this address >> 9] != DIRTY) 

           CARD_TABLE [this address >> 9] = DIRTY;

      先判断是不是已经是脏表了

GC相关参数

垃圾回收器

垃圾回收器是垃圾回收算法的实践者。

接下来是7种不同的hotspot收集器

新生代：Serial,parNew,Parallel Scavenge,G1

老年代：CMS ，Serial Old(MSC)，Parallel Old,G1

1，Serial/Serial Old

Serial:新生代，单线程，串行，标记清除算法

  • 会阻塞用户线程

Serial Old：配合Serial使用，作为老年代垃圾收集器，单线程，串行，标记整理算法

  • 阻塞用户线程

2，ParNew(Serial的并行版本)

新生代，标记复制算法，多线程，

  • 除了多线程并行收集以外，其他和Serial差别不大

  • 是CMS(一个老年代收集器)开启后默认开启的新生代收集器

  • 阻塞用户线程

3，Parallel Scavenge

新生代，标记复制算法，多线程，

  • 吞吐量优先，尽量缩短垃圾收集占用的总时间的比值，除此之外和ParNew差不多

  • 吞吐量=运行代码时间/(运行代码时间+垃圾收集时间)

  • 一个和ParNew的区分就是它能自动调节

4，Parallel Old(Parallel Scavenge的老年代版本)

老年代，标记整理算法，其他和Parallel Scavenge差不多

  • 阻塞用户线程
  • 在注重吞吐量或资源稀缺的情况下，可以考虑Parallel Scavenge/Parallel Old组合

5，CMS(Concurrent Mark Sweep)

老年代，多线程，并发标记，标记清除算法

  • 响应时间优先，单次stw的时间最短.

  • 初次标记stw，但是仅仅标记gcroots能关联到的对象，速度很快，会阻塞用户线程
  • 并发标记会变量整个对象图，时间较长但是是和用户线程并发执行，使用的是并发可达性分析的增量更新法。
  • 重新标记是标记那些在并发标记的同时用户线程产生的增量，会阻塞用户线程
  • 并发清除，清除线程和用户线程并发运行
    • 由于并发会降低总程序的吞吐量，所以出现和用户线程交替运行(i-cms，增量式并发收集器)这种变种,事实证明，效果很一般，jdk7已废弃.

缺点：

  • cms在并发清理的时候用户线程同时也在产生对象，产生的垃圾（浮动垃圾）只能在下一次才能回收掉，所以需要预留足够的空间给用户线程使用，如果预留的空间不足以满足并发的时候的需要就会并发失败(Concurrent Mode Failure),临时启用Serial Old收集器（串行，会阻塞用户线程）
  • CMS基于标记清除算法，会产生内存碎片，给的解决方案是，在若干次fullGC后，先整理碎片，再回收。

6,G1收集器（garbage first)

在jdk9，G1已经替代了parallel Scavebge/parallel Old组合，Cms完全被标记为(Deprecate)不推荐使用，并成为jdk9的默认垃圾收集器

内存布局的变化：

  • 把堆内存分为若干个Region,，每一个Region都可以是新生代，幸存区，或者是老年代的其中一个。

面向堆的任何部分组成回收集：

  • 以前的回收器有minorGC,majorGC,fullGC，分别回收新生代，老年代，全部。G1的Mixed GC模式，可以面向堆的任何部分组成回收集。
  • G1的收集器会去追踪各个Region里面的堆积的垃圾的”价值”大小，然后在后台维护一个优先级列表，根据用户允许的停顿时间，价值大的先回收(garbage first名字的由来)，这样就尽可能保证了手机效率。

G1面临的问题：

跨代引用如何实现？

  • G1的记忆集是个哈希表，key是 别的存在引用我们的Region的起始地址，Value是一个集合，里边是卡表的索引号。需要写前屏障和写后屏障一起配合。
  • G1记忆集的复杂性会导致需要较多堆空间（10——20%）来维护记忆集工作。

G1运行流程：

​ 1，初始标记，仅仅是标记一下GC roots能直接关联到的对象，并修改TAMS指针的值，使Region中的并发分配区域可用。虽然初始标记需要阻塞用户线程(时间很短)，但是G1借助和minorGc同步运行此阶段，所以此阶段并没有额外的停顿。

  • TAMS指针：回收过程中用户线程还是会分配对象，G1在每个Region中设置了两个TAMS指针专门划分区域供并发时分配对象使用。

​ 2，并发标记，从gcroots开始进行并发可达性分析，使用的是原始快照法。

​ 3，最终标记，处理并发标记原始快照遗留下的SATB记录（记录了并发时有引用变动的对象），会阻塞用户线程

​ 4，筛选回收，基于标记=复制，把存活的对象复制到新的region中，旧的Region清空。

  • 注意，宏观上看，G1使用的时标记整理算法，但是微观Region之间采用的是标记复制算法。和CMS的标记清除不一样，不会产生内存碎片。

实战

1，大对象直接进入老年代

2，当一个类的所有实例都不再存货并且类所在的加载器的所有类都没被使用，就可以卸载此类，一般框架中会有需求

3，Region中有一类特殊的Hunongous Region专门用来存储大对象，对象容量超过了Region的一般就判断为大对象，当对象超过了region大小会存储在连续的的Hunongous Region里，G1优先回收大对象，并且不会复制。
